## <a name="update-target-framework-moniker-tfm"></a>Aktualisieren des Zielframeworkmonikers (Target Framework Moniker, TFM)

Für Projekte, die auf .net Core abzielen, sollte der [TFM](/dotnet/standard/frameworks) einer Version größer oder gleich .net Core 2,2 verwendet werden. Aktualisieren Sie in der Projektdatei den `<TargetFramework>` inneren `netcoreapp2.2`Text des Knotens wie folgt:

```xml
<TargetFramework>netcoreapp2.2</TargetFramework>
```

Projekte, die auf .NET Framework abzielen, verwenden möglicherweise weiterhin den TFM einer Version, die größer oder gleich .NET Framework 4.6.1 ist:

```xml
<TargetFramework>net461</TargetFramework>
```

## <a name="adopt-the-iis-in-process-hosting-model"></a>Übernehmen des in-Process-Hostingmodells von IIS

Um das [in-Process-Hostingmodell für IIS](xref:host-and-deploy/iis/index#in-process-hosting-model)zu über `<AspNetCoreHostingModel>` nehmen, fügen Sie die `InProcess` -Eigenschaft `<PropertyGroup>` mit dem Wert zu einem in der Projektdatei hinzu:

```xml
<AspNetCoreHostingModel>InProcess</AspNetCoreHostingModel>
```

Das in-Process-Hostingmodell wird nicht für ASP.net Core Apps unterstützt, die auf .NET Framework abzielen

Weitere Informationen finden Sie unter <xref:host-and-deploy/aspnet-core-module#hosting-models>.

## <a name="update-package-references"></a>Aktualisieren von Paketverweisen

Wenn .net Core als Ziel verwendet wird, entfernen Sie das `Version` -Attribut des metapaketverweises in der Projektdatei. Die Einbindung eines- AttributsführtzufolgenderWarnung:`Version`

```console
A PackageReference to 'Microsoft.AspNetCore.App' specified a Version of `2.2.0`. Specifying the version of this package is not recommended. For more information, see https://aka.ms/sdkimplicitrefs
```

Weitere Informationen finden Sie unter <xref:fundamentals/metapackage-app>.

Der metapaketverweis sollte dem folgenden `<PackageReference />` Knoten ähneln:

```xml
<ItemGroup>
  <PackageReference Include="Microsoft.AspNetCore.App" />
</ItemGroup>
```

Wenn Sie auf .NET Framework abzielen, aktualisieren Sie das `Version` Attribut jedes Paket Verweises auf 2.2.0 oder höher. Im folgenden finden Sie die Paket Verweise in einem typischen ASP.net Core 2,2-Projekt, das auf .NET Framework abzielt:

```xml
<ItemGroup>
  <PackageReference Include="Microsoft.AspNetCore" Version="2.2.0" />
  <PackageReference Include="Microsoft.AspNetCore.CookiePolicy" Version="2.2.0" />
  <PackageReference Include="Microsoft.AspNetCore.HttpsPolicy" Version="2.2.0" />
  <PackageReference Include="Microsoft.AspNetCore.Mvc" Version="2.2.0" />
  <PackageReference Include="Microsoft.AspNetCore.StaticFiles" Version="2.2.0" />
</ItemGroup>
```

Wenn Sie auf das [Microsoft. aspnetcore. Razor. Design](https://www.nuget.org/packages/Microsoft.AspNetCore.Razor.Design/) -Paket verweisen, `Version` aktualisieren Sie das zugehörige-Attribut in 2.2.0 oder höher. Wenn dies nicht der Fall ist, führt dies zu folgendem Fehler:

```console
Detected package downgrade: Microsoft.AspNetCore.Razor.Design from 2.2.0 to 2.1.2. Reference the package directly from the project to select a different version.
```

## <a name="update-kestrel-configuration"></a>Kestrel-Konfiguration aktualisieren

Wenn die APP aufruft <xref:Microsoft.AspNetCore.Hosting.WebHostBuilderKestrelExtensions.UseKestrel*> , `CreateDefaultBuilder` indem Sie in der `Program` [createwebhostbuilder-Methode](xref:fundamentals/host/web-host#set-up-a-host) der-Klasse `ConfigureKestrel` aufruft, rufen Sie auf, um den `UseKestrel` Kestrel-Server anstelle von zu konfigurieren, um Konflikte mit IIS zu vermeiden. [ in-Process-Hostingmodell](xref:host-and-deploy/iis/index#in-process-hosting-model):

```csharp
public static IWebHostBuilder CreateWebHostBuilder(string[] args) =>
    WebHost.CreateDefaultBuilder(args)
        .UseStartup<Startup>()
        .ConfigureKestrel((context, options) =>
        {
            // Set properties and call methods on options
        });
```

Wenn die APP nicht aufruft `CreateDefaultBuilder` und den Host manuell in der `Program` -Klasse erstellt, <xref:Microsoft.AspNetCore.Hosting.WebHostBuilderKestrelExtensions.UseKestrel*> rufen Sie `ConfigureKestrel` **vor** dem Aufrufen von auf:

```csharp
public static void Main(string[] args)
{
    var host = new WebHostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .UseKestrel()
        .UseIISIntegration()
        .UseStartup<Startup>()
        .ConfigureKestrel((context, options) =>
        {
            // Set properties and call methods on options
        })
        .Build();

    host.Run();
}
```

## <a name="update-cors-policy"></a>Aktualisieren der cors-Richtlinie

In ASP.net Core 2,2 antwortet die cors-Middleware mit einem Platzhalter Ursprung`*`(), wenn eine Richtlinie einen beliebigen Ursprung zulässt und Anmelde Informationen zulässt. Anmelde Informationen werden nicht unterstützt, wenn`*`ein Platzhalter Ursprung () angegeben wird, und Browser die cors-Anforderung nicht zulassen. Weitere Informationen, einschließlich der Optionen zum Beheben des Problems auf dem Client, finden Sie in der [MDN](https://developer.mozilla.org/docs/Web/HTTP/CORS/Errors/CORSNotSupportingCredentials)-Webdokumentation.

Um dieses Problem auf dem Server zu beheben, führen Sie eine der folgenden Aktionen aus:

* Ändern Sie die cors-Richtlinie, um keine Anmelde Informationen mehr zuzulassen. Das heißt, entfernen Sie den- <xref:Microsoft.AspNetCore.Cors.Infrastructure.CorsPolicyBuilder.AllowCredentials*> Befehl, wenn Sie die Richtlinie konfigurieren.
* Wenn Anmelde Informationen erforderlich sind, damit die cors-Anforderung erfolgreich ist, ändern Sie die Richtlinie so, dass zulässige Hosts angegeben werden. Verwenden `builder.WithOrigins("https://api.example1.com", "https://example2.com")` Sie<xref:Microsoft.AspNetCore.Cors.Infrastructure.CorsPolicyBuilder.AllowAnyOrigin*>z. b. anstelle von.

## <a name="update-logging-code"></a>Protokollierungs Code aktualisieren

Der empfohlene Protokollierungs Konfigurations Code hat sich nicht von 2,1 zu 2,2 geändert, aber einige 1. x-Codierungs Muster, die weiterhin in 2,1 funktionieren, funktionieren nicht mehr in 2,2.

Wenn Ihre APP die Initialisierung, Filterung und Konfiguration von `Startup` `Program.Main`Anbietern in der-Klasse protokolliert, verschieben Sie den folgenden Code in:

* Anbieter Initialisierung:

  1. x-Beispiel:

  ```csharp
  public void Configure(IApplicationBuilder app, ILoggerFactory loggerFactory)
  {
      loggerFactory.AddConsole();
  }
  ```

  2,2 Beispiel:

  ```csharp

  public static void Main(string[] args)
  {
      var webHost = new WebHostBuilder()
          // ...
          .ConfigureLogging((hostingContext, logging) =>
          {
              logging.AddConsole();
          })
          // ...
  }
  ```

* Filterung

  1. x-Beispiel:

  ```csharp
  public void Configure(IApplicationBuilder app, ILoggerFactory loggerFactory)
  {
      loggerFactory.AddConsole(LogLevel.Information);
      // or
      loggerFactory.AddConsole((category, level) => 
          category == "A" || level == LogLevel.Critical);
  }
  ```

  2,2 Beispiel:

  ```csharp
  public static void Main(string[] args)
  {
      var webHost = new WebHostBuilder()
          // ...
          .ConfigureLogging((hostingContext, logging) =>
          {
              logging.AddConsole()
                     .AddFilter<ConsoleLoggerProvider>
                         (category: null, level: LogLevel.Information)
                     // or
                     .AddFilter<ConsoleLoggerProvider>
                         ((category, level) => category == "A" ||
                             level == LogLevel.Critical)
              );
          })
          // ...
  }
  ```

* Das Laden der Konfiguration:

  1. x-Beispiel:

  ```csharp
  public void Configure(IApplicationBuilder app, ILoggerFactory loggerFactory)
  {
      loggerFactory.AddConsole(Configuration);
  }
  ```

  2,2 Beispiel:

  ```csharp
  public static void Main(string[] args)
  {
      var webHost = new WebHostBuilder()
          // ...
          .ConfigureLogging((hostingContext, logging) =>
          {
              logging.AddConfiguration(hostingContext.Configuration.GetSection("Logging"));
              logging.AddConsole();
          })
          // ...
  }
  ```

Weitere Informationen finden Sie unter <xref:fundamentals/logging/index>
